# 工作流系统

工作流是 Kaloscope 的核心能力。

## 设计目标

项目通过工作流表达站点接入和自动化逻辑，核心目标包括：

- 将资源搜索与刮削逻辑从程序代码中解耦
- 允许用户通过拖拽节点快速搭建自动化流程
- 让社区可以共享、复用和持续维护工作流模板

## 已知能力

根据 README，当前工作流系统具备以下特性：

- 基于节点的可视化工作流编辑器
- 支持 HTTP 请求、Python 脚本、条件分支、循环控制等节点类型
- 支持从 GitHub 仓库导入社区工作流模板
- 支持定时触发与计划执行

## 节点结构

后端代码中的节点注册系统显示，工作流节点被划分为四个基础分组：

- `START`：流程入口类节点
- `GENERAL`：通用处理类节点
- `CONTROL`：条件与循环控制类节点
- `END`：流程出口类节点

## 节点运行方式

从节点基类可以确认以下实现细节：

- 每个节点会注册一个 `node_type`、图标、分组和顺序
- 节点会声明字段与连接句柄，供前端编辑器渲染
- 运行时通过统一执行器分发到对应节点实现
- 节点输出会以快照形式写回上下文，用于继续向后传递

这意味着前端画布和后端执行器之间使用的是一套统一的节点 schema。

## 典型应用场景

- 从资源站搜索番剧并解析详情页
- 根据关键词定时检索新资源并自动下发到下载器
- 调用脚本节点处理页面内容或自定义过滤规则
- 将媒体文件入库前执行一组清洗、匹配和命名步骤

## 一个常见流程示例

下面是一条更贴近项目定位的典型链路：

1. `schedule` 或 `search` 节点作为入口。
2. `http` 节点请求资源站接口或网页。
3. `text` 或 `script` 节点解析返回内容、提取结果。
4. `condition` 节点过滤标题、分辨率、字幕等条件。
5. `loop` 节点遍历候选资源。
6. `details` 或 `ingest` 节点进入详情和入库流程。
7. `end` 节点将结果交给搜索展示、计划任务或下载模块。

这部分后续可以继续补成可视化示意图或节点级教程。

## 安全提示

工作流可能包含任意代码或外部请求。导入第三方工作流前，应重点检查：

- 请求目标与认证信息用途
- Python 脚本节点的执行行为
- 是否会写入本地敏感目录
- 是否会将数据上传到外部服务
